# 基于 RocketMQ-A2A 和 LiteTopic 的智能体编排中枢设计文档

## 1. 项目概述

**Orchestrator** 是一个具备编排能力的 **A2A 标准智能体**。它本身符合 Google A2A 协议，对外表现为一个可被调用的 Agent（拥有 `AgentCard`、接受 `Task`、返回 `Artifact`），但其内部具备独特的编排中枢能力：通过大语言模型进行 **ReAct 风格的循环编排**（规划 → 执行 → 观察 → 再规划），将用户任务拆解分发给多个子智能体执行，并基于 **Apache RocketMQ-A2A** 实现跨智能体的异步协作与状态共享。

作为 Agent，它与普通业务智能体的区别在于：
- **普通 Agent**：接收任务 → 自身执行 → 返回结果
- **编排中枢 Agent**：接收任务 → LLM 拆解为子任务 → 分发给多个子 Agent 并行/顺序执行 → 收集结果 → 再规划或汇总 → 返回最终结果

系统的核心创新在于：利用 RocketMQ-A2A 的 LiteTopic 将每次会话建模为**可重放的事件流**，使得多智能体协作中的记忆共享、断点续传和状态透明成为可能。

## 2. 核心技术选型

| 组件 | 选型                        | 说明 |
|------|---------------------------|------|
| **通信协议** | Google A2A Protocol v1.0  | Agent-to-Agent 行业标准协议，支持 Task、Message、Artifact、Streaming 等完整语义 |
| **消息中间件** | Apache RocketMQ-A2A       | 基于 Apache RocketMQ 的 A2A 协议实现，提供 LiteTopic 会话路由模型 |
| **消息队列** | Apache RocketMQ 5.x       | 生产级消息中间件，支持消息持久化、顺序消费、负载均衡 |
| **LLM 编排** | AgentScope-Java           | 多智能体编排框架，提供 Agent 抽象、Memory 管理和工作流支持 |
| **角色设计** | 参考 ohmyopenagent sisyphus | Hub 采用 sisyphus 角色模式：循环推理、任务拆解、结果汇聚、动态再规划 |
| **服务注册与发现** | RocketMQ Topic 订阅机制       | 子智能体通过订阅能力 Topic 实现轻量级自注册，Hub 通过 Topic 寻址分发任务 |

## 3. 设计理念

### 3.1 协议标准化
外部访问者通过标准的 **A2A 协议**与中枢交互。A2A 定义了完整的任务生命周期（`SUBMITTED` → `WORKING` → `COMPLETED/FAILED/CANCELED`），支持同步响应、流式推送、任务取消和中断输入，天然适配多智能体协作场景。

### 3.2 ReAct 循环编排
LLM 不是一次性生成完整计划后执行到底，而是采用 **ReAct（Reasoning + Acting）循环**：
- **Reason**：基于当前会话上下文，推理用户意图并生成任务拆解计划（DAG）
- **Act**：将子任务分发给对应智能体执行
- **Observe**：收集各子任务的执行结果和中间输出
- **Re-plan**：基于观察结果，决定任务完成或生成下一轮 DAG

每轮循环的决策计划（OrchestrationPlan）和执行摘要（TaskSummary）均作为事件写入会话事件流，形成可追溯的编排决策链。

### 3.3 消息驱动解耦
所有智能体（包括中枢自身）均通过发布和订阅 Topic 进行协作，实现时间和空间的完全解耦：
- **功能性 Topic**：用于子任务命令的分发和能力寻址（如 `/agents/weather`）
- **会话事件流**：用于会话内记忆共享和状态同步，基于 LiteTopic 实现会话隔离

### 3.4 会话即事件流
每个任务会话对应一个 **LiteTopic** 标识的事件流。所有参与协作的智能体和中枢将以下事件追加到该流：
- `OrchestrationDecisionEvent`：Hub 的 LLM 推理决策和生成的 DAG
- `TaskCommandEvent`：分发给子智能体的任务命令
- `TaskProgressEvent`：子智能体的执行进度、中间结果、环境感知
- `TaskSummaryEvent`：子任务完成后的结果摘要和关键发现
- `SessionSnapshotEvent`：Hub 生成的会话状态快照（用于故障恢复）

任何组件均可通过重放该事件流重建完整的会话上下文。

## 4. 系统架构

系统架构由三层组成：中枢层、消息中间件层、子智能体层。

### 4.1 架构拓扑

```text
外部客户端
|
| (A2A 协议：tasks/send, tasks/cancel, tasks/get)
v
编排中枢 (Hub)
- A2A 协议适配层（A2A Server）
- ReAct 编排引擎（LLM 规划 → 任务调度 → 结果汇聚 → 再规划）
- 上下文管理层（事件流重放、快照生成、记忆分层）
|
| 1. A2A 协议通信（请求/响应/流式推送）
| 2. 会话事件流（记忆事件追加与消费）
v
RocketMQ-A2A 消息中间件
- 功能性 Topic (如 /agents/weather, /agents/hotel)
  → 子任务命令分发，支持负载均衡
- 会话事件 Topic (绑定 LiteTopic 做会话隔离)
  → 记忆事件流，支持重放恢复
|
| 子智能体订阅功能性 Topic 获取任务命令
| 子智能体向会话事件流写入执行结果和进度
| 中枢订阅会话事件流收集结果并做下一轮决策
v
子智能体集群
- 天气智能体 (订阅 /agents/weather)
- 酒店智能体 (订阅 /agents/hotel)
- 等任意符合 A2A 协议的业务智能体
```

### 4.2 通信模式说明

**A2A 协议通信（协议层）**：
- 外部客户端通过 A2A 协议与 Hub 交互（HTTP/JSON-RPC）
- Hub 通过 A2A 协议向子智能体发送任务（基于 RocketMQTransport）
- 子智能体通过 A2A 协议向 Hub 返回任务状态和结果
- LiteTopic 在 A2A 层用于**异步响应的路由**：请求方订阅 `sessionId` 对应的 LiteTopic，响应方将结果发送到该 LiteTopic

**会话事件流（记忆层）**：
- 与 A2A 协议通信解耦，独立的记忆事件通道
- 所有组件（Hub + 子智能体）均可向该流追加事件
- 所有组件均可从该流消费事件，重建上下文
- 基于 RocketMQ 消息持久化保障，支持断点重放

## 5. 核心模块设计

### 5.1 编排中枢

- **A2A 协议适配层**：实现 A2A 服务端（`AgentCard`、`Task`、`Message` 等标准实体），将外部 A2A 请求转换为内部事件，将内部执行结果封装为 A2A 响应
- **ReAct 编排引擎**：
  - **规划器 (Planner)**：基于当前会话上下文，调用 LLM 生成包含子任务、依赖关系和目标智能体能力描述的编排计划（DAG）
  - **调度器 (Scheduler)**：根据 DAG 将子任务封装为 A2A 消息发送到对应的功能性 Topic
  - **汇聚器 (Aggregator)**：收集各子任务的执行结果，组装为下一轮推理的观察输入
  - **再规划器 (Re-planner)**：基于观察结果，判断任务完成或生成下一轮 DAG
- **上下文管理层**：订阅会话事件流，维护当前会话的完整状态；生成 `SessionSnapshotEvent` 和 `OrchestrationDecisionEvent`

### 5.2 RocketMQ-A2A 与会话事件流

会话事件流是短期记忆的载体。一次用户会话对应一个 LiteTopic 标识的事件流，具备以下特性：

- **会话隔离**：每个 `session-id` 对应独立的 LiteTopic，不同会话的事件流互不干扰
- **事件流持久化**：所有记忆事件按时间顺序追加，基于 RocketMQ 消息持久化存储
- **可重放恢复**：任何组件因故障重启后，可通过重放事件流从任意已知 offset 恢复会话上下文
- **广播与共享**：Hub 和所有参与该会话的子智能体均可向事件流追加事件，也均可消费事件流重建状态
- **功能寻址**：子智能体能力的发现通过功能性 Topic 订阅实现（如 `/agents/weather`），与事件流解耦

### 5.3 上下文管理：与普通 Agent 的核心区别

编排中枢作为 Agent，同样需要记忆管理。但其记忆管理的独特之处在于**多智能体共享**和**编排决策链的可追溯**：

| 维度 | 普通 Agent | 编排中枢 Agent |
|------|-----------|---------------|
| **记忆范围** | 单 Agent 内部对话历史 | 跨多个子 Agent 的完整协作事件流 |
| **记忆共享** | 无共享需求 | Hub 和所有子 Agent 共用同一会话事件流 |
| **记忆内容** | 用户消息 + Agent 回复 | 编排决策、任务命令、执行进度、结果摘要、快照 |
| **恢复方式** | 重启后丢失（除非持久化） | 重放 LiteTopic 事件流即可重建完整上下文 |
| **决策追溯** | 无 | 每轮 ReAct 决策均写入事件流，支持审计 |

**简要说明**：
- **会话事件流**：以 LiteTopic 为标识，所有参与者（Hub + 子 Agent）均可追加和消费事件
- **事件类型**：编排决策、任务命令、执行进度、任务摘要、会话快照
- **快照机制**：在编排决策后和任务完成后自动生成轻量快照，用于故障恢复时减少重放长度
- **长期记忆**：会话结束后事件流可归档至外部存储，支持历史检索（通用能力，此处不展开）

### 5.4 子智能体 (Sub-Agent)

子智能体是符合 A2A 标准的普通业务 Agent，通过功能性 Topic（如 `/agents/weather`）接收 Hub 分发的子任务。与普通 Agent 不同的是：
- **共享记忆**：子 Agent 可向会话事件流写入执行结果，也可消费事件流了解其他 Agent 的中间输出
- **无状态**：不持久化会话状态，故障后通过重放事件流恢复上下文
- **动态扩缩**：通过订阅/取消订阅功能性 Topic 实现注册与下线

## 6. 工作流程示例：智能差旅安排

### 6.1 ReAct 循环执行

**Round 1 —— 规划与执行**

1. 用户发送任务："帮我规划明天的出差，包括天气和酒店"
2. Hub ReAct 规划器生成 DAG：任务 A（天气）→ 任务 B（酒店，依赖 A）
3. 任务 A 分发给天气智能体，执行结果（"明天北京中雨"）和推理建议写入会话事件流

**Round 2 —— 观察与再规划**

4. Hub 观察到任务 A 完成，触发再规划，确认执行任务 B
5. 任务 B 分发给酒店智能体。酒店智能体通过重放事件流了解到天气结果，据此筛选酒店并完成预订

**Round 3 —— 完成**

6. Hub 观察到所有任务完成，汇总结果返回用户

### 6.2 用户打断

用户在任务 B 执行中发送新指令"别查酒店了，改查机票"：
- Hub 调用 A2A `tasks/cancel` 取消任务 B
- 生成中断决策事件，ReAct 再规划器基于新意图生成任务 C（机票）

### 6.3 故障恢复

Hub 进程崩溃重启后：
- 从会话事件流中找到最近的快照，加载状态
- 从快照 offset 之后增量重放事件，恢复到崩溃前的上下文

## 7. 优势总结

- **标准化协议**：采用 Google A2A 协议，无缝接入任何符合 A2A 标准的第三方智能体
- **ReAct 循环编排**：LLM 动态规划 + 观察反馈 + 再规划，适配复杂多变的协作场景
- **事件流式记忆**：基于 RocketMQ 持久化的事件流，实现记忆共享、断点恢复、状态透明
- **彻底异步解耦**：所有通信通过消息队列，智能体可独立开发、部署、伸缩
- **天然削峰填谷**：消息队列缓冲突发流量，保护下游智能体
- **用户可控**：A2A 原生支持任务取消和中断输入，用户可在任意时刻介入调整

## 9. 后续扩展

- **长期记忆归档**：会话结束后的完整事件流归档至对象存储或数据库，支持 LLM 检索历史会话
- **智能体市场**：基于功能性 Topic 的订阅机制，构建开放的子智能体注册、发现与能力匹配市场
- **多模态支持**：A2A 协议和 RocketMQ 均支持大消息体，可扩展至图片、音频、视频等多模态智能体协作
- **安全与鉴权**：A2A 协议认证机制落地、子智能体间消息签名、Hub 访问控制策略
- **人机协作增强**：支持人在循环（Human-in-the-loop）模式，在关键决策点暂停等待人工确认
